# 1 引言
## 1.1 编制目的
本报告用于对“基于 Web 的助农产品销售系统”的立项价值、技术路线、实施条件和推广前景进行分析，判断该项目是否适合作为本科毕业设计课题继续开展，并为后续需求规格说明、概要设计说明、数据库设计说明和系统实现提供依据。

本文档的预期读者包括指导教师、项目开发者、测试人员、论文评审人员以及后续阅读系统设计文档的相关人员。

## 1.2 项目背景与现状
在乡村振兴和“互联网+农业”持续推进的背景下，农产品线上销售已经成为促进农户增收和优化农产品流通的重要手段。但传统农产品销售模式仍然存在若干突出问题：

1. 农户销售渠道分散，获客能力弱，难以稳定触达城市消费群体。
2. 农产品标准化程度相对较低，消费者对来源、品质和安全性的信任建立成本较高。
3. 大型综合电商平台虽然交易能力成熟，但入驻流程、运营成本和规则复杂度较高，不完全适合中小农户和合作社。
4. 现有部分助农平台更偏信息展示，缺少订单、售后、审核、对账、溯源和平台治理等完整闭环。

基于上述背景，本项目拟建设一套面向助农场景的三端 Web 平台。系统以普通消费者、农户/合作社和平台管理员为主要角色，围绕商品展示、商品溯源、购物车、订单、模拟支付、售后协商、商家经营、平台审核、实时通知、日志审计、自动备份和滞销帮扶等模块展开。

结合当前项目代码实现情况，系统已经形成用户端、商家端、管理员端三端协同结构，前后端采用分离架构，后端已接入 JWT 认证、管理员权限控制、Redis 缓存、WebSocket 实时推送、定时任务、二维码溯源、日志和备份等能力，具备较强的开发落地基础。

# 2 可行性研究前提
## 2.1 业务前提
本项目面向助农电商业务场景，平台中的主要参与角色如下：

1. 游客和普通用户：浏览首页、商品、新闻、滞销专区，登录后可加入购物车、提交订单、完成模拟支付、查看物流、评价商品、发起售后与接收系统消息。
2. 商家：注册入驻、维护店铺资料、发布商品、填写溯源信息、处理订单、填写物流、处理售后、查看统计和对账、维护收款账户、发起提现申请。
3. 管理员：完成登录认证、看板查看、用户与商家管理、商品审核、评论管理、新闻管理、订单监管、售后裁决、打款台账、收款账户审核、提现审核、角色权限、日志审计、备份和滞销专区管理。

业务上，系统覆盖了“注册登录—上架审核—下单支付—订单履约—售后协商—平台介入—结算对账—运维审计”的完整闭环，研究目标明确，角色关系清晰，能够支撑毕业设计的业务完整度要求。

## 2.2 项目目标
本项目的目标可以从技术、业务、管理和研究四个层面进行概括。

1. 技术目标：采用 Vue 3 + Vite + Pinia + Element Plus 构建前端，采用 Spring Boot + MyBatis-Plus + MySQL + Redis 构建后端，形成结构清晰、易于扩展和维护的前后端分离系统。
2. 业务目标：围绕助农场景构建商品、订单、售后和溯源闭环，降低农户线上经营门槛，增强消费者对农产品来源与质量的信任。
3. 管理目标：为平台提供商家审核、商品审核、评论与内容管理、日志审计、权限控制、数据备份、资金监管等治理能力。
4. 研究目标：在本科毕业设计范围内，形成一套具有助农特色、工程结构完整、能够展示真实开发能力的 Web 软件系统。

## 2.3 条件、假设与限制
### 2.3.1 开发与运行条件
1. 开发环境为 Windows 11，开发工具包括 IntelliJ IDEA、VS Code、Maven、Node.js、Navicat 等。
2. 运行环境依赖 JDK 17、MySQL 8.x、Redis 6.x 和主流浏览器。
3. 前端通过浏览器访问，后端以 Spring Boot 服务形式运行，并通过 RESTful API 与 WebSocket 对外提供能力。

### 2.3.2 项目假设
1. 平台用户具备基本浏览器使用能力。
2. 农户或合作社愿意以较低门槛的方式维护商品和店铺资料。
3. 消费者对可溯源农产品和助农场景具有一定接受度。
4. 毕业设计周期内以实现核心闭环为主，外部第三方能力采用模拟实现或预留扩展方式处理。

### 2.3.3 现实限制
1. 真实支付、真实短信网关、真实物流轨迹和对象存储当前未完全接入，支付与短信校验属于模拟或轻量化实现。
2. 系统以本科毕业设计为目标，优先满足功能完整性、结构规范性和可演示性，不以大规模商用部署为直接目标。
3. 数据规模和并发压力主要以课程与答辩场景为主，性能设计采取“轻量缓存 + 合理索引 + 原子库存扣减 + 定时任务 + 实时推送”的方式。

## 2.4 决定可行性的主要因素
本项目是否可行，主要取决于以下因素：

1. 选题是否具有明确现实意义并与助农电商场景高度匹配；
2. 所采用技术栈是否成熟、稳定，能否支撑三端业务和论文展示；
3. 核心业务流程是否能够在毕业设计周期内完成闭环实现；
4. 系统运行与维护成本是否处于本科毕业设计可接受范围内；
5. 当前代码实现程度是否足以支撑后续撰写需求、设计、测试和论文正文。

# 3 所建议系统的可行性分析
## 3.1 建议系统说明
本系统是一套基于 B/S 架构和前后端分离模式构建的助农产品销售平台。前端为单页应用，按用户端、商家端和管理员端进行页面组织；后端按用户、商家、管理员和公共服务进行模块划分；数据库使用 MySQL 保存业务数据，Redis 负责缓存和部分实时辅助数据，WebSocket 用于三端实时通知。

在方案选型上，本项目主要比较了以下三种实现思路：

| 方案 | 特点 | 适用性分析 |
| ---- | ---- | ---- |
| 传统线下销售或信息登记系统 | 实现简单，但缺少线上交易和闭环管理能力 | 无法体现软件工程课题完整度，不适合作为本课题最终方案 |
| 依托通用大型电商平台二次运营 | 平台能力成熟，但入驻、规则和运营成本较高 | 难以体现自主设计实现，也不利于突出助农特色 |
| 自主设计三端 Web 助农电商平台 | 可按课题目标实现溯源、审核、售后、对账与治理闭环 | 最符合本科毕业设计的工程性、研究性和展示性要求 |

综合比较后，自主设计并实现一套面向助农场景的三端 Web 系统，是当前条件下最合适的方案。

当前代码中已经落地的关键能力包括：

1. 三端独立登录与业务页面；
2. JWT 认证与登录类型拦截；
3. 管理员角色权限控制与权限缓存；
4. 商品、订单、售后、评论、新闻、滞销专区等核心业务；
5. 商品溯源信息维护、独立溯源档案页与二维码生成；
6. Redis 公共数据缓存和热门商品排行；
7. WebSocket 实时消息刷新机制；
8. 订单超时取消、自动打款、自动备份、提现打款重试等定时任务；
9. 系统日志、审核记录和数据备份。

因此，建议系统并非概念化设计，而是已经具备主要实现基础的工程化原型。

从工程规模看，系统当前已具备较完整的实现体量，主要表现为：

1. 前端存在用户端、商家端、管理员端三套路由与页面；
2. 后端已形成用户、商家、管理员、公共服务四类控制器分组；
3. 当前数据库实际共有 29 张表，其中 27 张为核心业务表，2 张为运行时自动初始化的辅助消息表；
4. 系统包含 4 个定时任务，分别用于订单超时取消、自动打款、自动备份和提现打款重试；
5. 系统包含 2 条 WebSocket 通道，分别面向订单沟通和统一实时刷新。

## 3.2 业务可行性分析
### 3.2.1 业务问题与系统功能的匹配度较高
本系统围绕农产品线上交易场景中的典型问题展开设计，例如信息不透明、交易流程割裂、售后沟通不及时、平台监管薄弱等。系统通过商品溯源、平台审核、售后协商、日志留痕和滞销帮扶等能力，对这些问题进行了较有针对性的响应。

### 3.2.2 三端协同能够形成完整闭环
从业务流程看，系统已经实现以下闭环：

1. 商家注册并提交资料；
2. 管理员审核商家与商品；
3. 用户浏览商品并下单；
4. 商家发货，用户收货和评价；
5. 用户发起售后，商家处理，必要时管理员介入；
6. 平台自动打款并生成对账记录；
7. 管理员查看日志、执行备份和管理后台内容。

该闭环覆盖了电商平台的关键业务链路，能够满足毕业设计对“业务完整性”的要求。

### 3.2.3 助农特色具有明确研究价值
相比通用电商系统，本系统强调以下特色：

1. 农产品独立溯源档案页与“一物一码”展示；
2. 按生鲜果蔬、粮油副食、干货特产、畜禽肉蛋组织分类差异化溯源字段；
3. 商家经营门槛低，适合农户/合作社；
4. 滞销专区与帮扶推荐；
5. 平台审核与内容治理；
6. 对账、补贴、提现等面向商家经营场景的能力。

这些特色使项目具有较强的主题鲜明性和论文研究价值。

## 3.3 技术可行性分析
### 3.3.1 技术栈成熟稳定
项目所采用的技术栈均为当前常见且资料丰富的成熟技术：

1. 前端：Vue 3、Vite、Pinia、TypeScript、Element Plus；
2. 后端：Spring Boot、MyBatis-Plus、Lombok、JWT、Spring WebSocket；
3. 数据层：MySQL、Redis；
4. 辅助能力：Swagger/OpenAPI、文件上传、二维码生成、定时任务。

这些技术具有学习资料多、社区活跃、实现难度适中的特点，适合作为本科毕业设计的主要实现基础。

### 3.3.2 当前系统结构清晰
从代码结构上看，系统已经形成较规范的分层组织方式：

1. 前端按照 `router`、`layout`、`views`、`apis`、`stores`、`utils` 分层组织；
2. 后端按照 `controller`、`service`、`mapper`、`entity`、`config`、`job`、`websocket` 分层组织；
3. 权限、日志、缓存、实时推送和定时任务被拆分为独立模块，便于后续维护与扩展。

### 3.3.3 关键技术点已经具备真实实现
当前代码中已经具备以下关键实现，不再停留于理论设计：

1. JWT 令牌登录；
2. 用户、商家、管理员三类身份校验；
3. 管理员权限拦截与角色权限缓存；
4. Redis 公共数据缓存和热门商品排行；
5. WebSocket 实时刷新三端消息与页面状态；
6. 订单超时取消、库存回滚与原子库存扣减；
7. 自动打款、自动备份、提现打款重试；
8. 日志审计与审核记录；
9. 独立溯源档案页、一物一码与溯源二维码生成；
10. 分类扩展溯源字段建模。

### 3.3.4 未完全接入部分对可行性影响可控
当前系统仍有部分能力采用模拟或预留扩展方式，例如模拟支付、轻量短信实现和未接入真实物流平台。但这些能力并不影响系统完成核心业务闭环，也不会阻碍毕业设计答辩展示，因此属于可接受范围内的技术折中。

此外，从当前项目状态看，前端已可执行 `npm run build` 完成生产构建，后端已可执行 `mvn -DskipTests compile` 完成编译校验，说明系统不仅具备设计上的技术可行性，也具备当前工作区层面的实现可交付性。

综上，本项目技术路线成熟，当前代码实现程度较高，整体技术可行性良好。

## 3.4 运行与维护可行性分析
### 3.4.1 运行环境要求适中
本系统对客户端要求较低，用户只需通过常见浏览器访问即可；服务端采用单体 Spring Boot 应用加 MySQL 和 Redis 的轻量部署方式，适合毕业设计、课程展示和小规模试运行场景。

### 3.4.2 系统具备基础运维能力
当前项目已经具备若干基础运维能力：

1. 系统日志和审核记录；
2. 每日自动备份和手动备份/恢复；
3. 订单超时与自动打款定时任务；
4. 异常订单、资金流转与提现审核页面；
5. 权限控制与后台访问边界。

这说明系统不仅能“跑起来”，也考虑到了基本运维与平台治理需求。

此外，系统在运行维护层面还具备以下特点：

1. 公共数据缓存和权限缓存可降低重复查询压力；
2. 关键业务通过实时推送减少人工刷新和状态滞后；
3. 订单、售后、打款和提现等核心流程均具备较清晰的状态流转；
4. 备份文件以 SQL 形式生成，便于答辩演示和轻量恢复。

### 3.4.3 维护复杂度可控
由于系统采用单体后端和前后端分离结构，模块边界较明确，功能命名较直观，维护复杂度处于本科毕业设计可接受范围。对于论文答辩和后续扩展而言，该架构既能体现完整性，也不会因过度复杂而难以说明。

## 3.5 项目风险与对策分析
虽然项目总体可行，但在毕业设计实施和答辩展示阶段仍存在若干现实风险，需要提前识别并控制。

### 3.5.1 外部能力未完全真实接入风险
当前支付、短信验证码和物流轨迹等能力仍以模拟实现或轻量方式为主，若在文档中表达不清，容易被误解为已经完成真实生产对接。

应对方式如下：

1. 在需求、设计、测试和论文正文中统一表述为“模拟实现”或“预留扩展接口”；
2. 在答辩演示时重点展示系统闭环和工程设计，而不夸大第三方真实接入程度；
3. 将第三方真实接入列为后续优化方向，而非当前完成目标。

### 3.5.2 答辩环境依赖风险
系统依赖 MySQL、Redis、后端服务和前端服务共同运行，若答辩前环境准备不充分，可能影响展示稳定性。

应对方式如下：

1. 固化演示账号、演示商品、演示店铺和溯源码数据；
2. 在答辩前完成一轮完整启动检查和关键路径联调；
3. 保留本地数据库备份和固定演示路径，降低现场临时变动风险。

### 3.5.3 边界操作与重复提交风险
电商系统中的支付、取消订单、商家发货、管理员退款等动作容易在重复点击、连续刷新或快速切页时暴露边界问题。

应对方式如下：

1. 前端关键按钮增加提交中锁定、防连点和状态提示；
2. 后端对支付、退款、库存扣减和售后状态流转执行幂等或状态校验；
3. 在测试报告中专门纳入高频页面和边界场景验证。

### 3.5.4 前端包体积偏大的性能风险
当前前端虽可成功构建，但构建输出仍存在较大 chunk 警告，说明在资源组织上仍有进一步优化空间。

应对方式如下：

1. 当前阶段以答辩可运行和功能闭环为优先目标；
2. 后续可通过动态导入、手动分包和资源裁剪进一步优化首屏加载体验；
3. 在论文中可将其作为“后续性能优化方向”进行说明。
