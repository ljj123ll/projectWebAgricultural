## 1.3 缺陷分级
- 一级缺陷：核心业务无法完成，系统无法继续运行或主要模块整体不可用。
- 二级缺陷：主要功能存在明显错误，虽然系统可运行，但业务闭环被破坏。
- 三级缺陷：功能基本可运行，但在特定输入、异常场景或边界条件下表现不符合预期。
- 四级缺陷：不影响核心业务完成，但存在界面、提示、兼容性或易用性问题。
- 五级缺陷：文案、样式或非关键辅助功能的轻微问题。

# 2 测试对象概述
## 2.1 系统简介
本系统为面向助农电商场景的三端 Web 平台，采用前后端分离架构实现。前端基于 Vue 3、Vite、Pinia 与 Element Plus 构建，后端基于 Spring Boot、MyBatis-Plus、MySQL 与 Redis 实现。

系统主要角色包括：

1. 用户：浏览商品、加入购物车、提交订单、完成模拟支付、查看物流、评价商品、申请售后、查看消息与溯源档案。
2. 商家：维护店铺信息、发布商品、填写基础溯源字段和分类特色溯源字段、生成二维码、处理订单、处理售后、查看统计、对账、补贴、收款账户与提现信息。
3. 管理员：管理用户、审核商家和商品、处理评论与新闻、查看订单、裁决售后、监管打款与提现、查看日志并执行备份恢复。

## 2.2 测试范围
本次测试覆盖以下范围：

1. 用户端：首页、商品列表、商品详情、商家主页、购物车、订单、售后、消息、溯源档案页。
2. 商家端：登录、店铺维护、商品发布、二维码生成、订单管理、售后处理、统计、对账、收款账户、提现。
3. 管理员端：登录、商家审核、商品审核、订单监管、售后裁决、资金流水、打款台账、日志与备份。
4. 公共能力：JWT 登录态、角色权限、Redis 缓存、WebSocket 实时刷新、订单沟通、定时任务、原子库存扣减。

不纳入本次重点测试的内容包括：

1. 第三方真实支付接口对接；
2. 真实短信平台接入；
3. 多节点分布式部署与压测；
4. 生产环境安全加固和容灾演练。

# 3 测试策略与方法
## 3.1 测试目标
本次测试的目标包括：

1. 验证系统三端核心业务流程是否完整闭环；
2. 验证项目答辩高频页面是否稳定、可演示、可解释；
3. 验证关键非功能能力是否具备基础实现，例如权限控制、实时刷新、缓存与定时任务；
4. 验证近期优化内容是否真实落地，包括独立溯源档案页、一物一码、分类特色溯源字段、原子库存扣减和商家/管理员高频页面容错处理。

## 3.2 测试方法
本报告采用以下测试方法：

1. 黑盒测试：从功能输入输出角度验证系统行为是否符合需求。
2. 场景测试：围绕“用户下单支付”“商家发布商品”“管理员审核与资金监管”等完整业务链路进行验证。
3. 构建测试：通过前后端编译与打包确认代码可构建。
4. 界面联调测试：结合本地页面访问和既有联调记录，验证核心页面是否可访问、可展示。
5. 代码核验测试：对库存扣减、溯源字段组织、WebSocket 通道和定时任务实现进行针对性检查。
6. 构建核验与轻量自动化测试：通过 Maven 编译、Vite 打包和 Spring Boot 上下文测试确认项目具备基础可运行性。

说明：本报告中的构建测试与代码核验结果来自本次实际执行，功能测试结果主要结合固定演示数据、本地联调情况和当前源码实现整理，适用范围以毕业设计答辩展示场景为主，不等同于生产环境大规模压测结论。

## 3.3 通过准则
系统通过测试应满足以下条件：

1. 核心业务流程能够完成，不出现一级或二级缺陷；

3. 前后端代码能够通过构建校验；

## 4.6 实时消息与状态刷新测试
### 4.6.1 测试分析
系统采用两条 WebSocket 通道：

1. `/ws/realtime`：用于订单、售后、审核、未读数等页面刷新事件；
2. `/ws/order-chat`：用于订单沟通正文传输，支持文本、图片、视频三种消息类型。

这种“双通道”设计避免了聊天消息与页面刷新事件混用，逻辑更清晰，也便于论文答辩中解释技术实现。

# 7 测试结论
通过本次测试可以得出以下结论：

1. 系统三端核心业务链路已经基本形成完整闭环，能够覆盖“商品浏览与溯源、下单支付、商家履约、售后处理、后台审核、资金监管”的主要场景。
2. 前端和后端工程均已通过构建校验，说明当前代码处于可编译、可部署状态。
3. 独立溯源档案页、一物一码、分类溯源字段、原子库存扣减、双 WebSocket 通道、自动打款与提现重试等关键设计点均已在当前项目中落地。
4. 商家端和管理员端高频演示页面经过稳定性优化后，答辩现场出现空白页、静默失败或关键数据缺失的风险已明显降低。
5. 本报告结论主要适用于当前本地部署、固定演示账号和答辩样例数据场景，不代表系统已经完成真实支付接入、生产级压测和多节点部署验证。

综合判断，当前“基于 Web 的助农产品销售系统”已达到毕业设计答辩所需的基本测试要求，具备论文撰写与现场展示的可支撑性。后续若继续完善，可进一步从真实支付接入、短信平台接入、自动化测试脚本和压测验证等方向深化。
